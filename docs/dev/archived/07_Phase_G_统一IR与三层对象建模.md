# 07_Phase_G_统一IR与三层对象建模.md

> **状态：✅ 完成（2026-04-13）**  
> G1~G6 全部完成，450 个单元测试通过，算子层主链无退化。

## 1. 阶段目标

本阶段目标是把当前偏算子层的表示与存储方式，整理为能够同时支撑算子层、模型层、应用层的统一研究底座。

阶段完成后，DeepMT 应具备以下能力：

- 同一套对象表示可以描述三层被测对象；
- 同一套 MR 记录结构可以承载不同来源、不同验证级别的关系；
- 同一套执行结果结构可以追踪框架、版本、运行环境、输入与证据；
- 在不破坏现有算子层闭环的前提下，为模型层、应用层和多框架扩展提供统一入口。

---

## 2. 本阶段要解决的问题

当前系统的主链已经围绕算子层形成闭环，但后续阶段将引入模型层、应用层和真实多框架适配。如果继续沿用当前偏算子层的表示方式，会出现以下问题：

- 新层次能力只能以补丁形式附着在现有模块上；
- 不同层的 MR、结果和证据难以统一管理；
- 跨框架实验与论文实验数据无法形成统一索引；
- 后续阶段会不断重复造新的结构与兼容逻辑。

因此，本阶段不是新增展示性功能，而是先把底座重构为论文后续开发的统一承载层。

---

## 3. 阶段边界

本阶段只做“统一表示、统一存储、统一追踪”的底座工作，不直接追求模型层或应用层功能的大规模上线。

本阶段不做的内容包括：

- 不在本阶段完成模型层 MR 自动生成；
- 不在本阶段完成应用层语义 MR 管线；
- 不在本阶段扩展新的复杂 Web 页面；
- 不对现有算子层流程做破坏性重写。

本阶段的原则是：**先完成统一承载能力，再接后续层次能力。**

---

## 4. 任务拆分

### G1：统一被测对象抽象 ✅

目标：建立三层统一对象抽象，明确算子、模型、应用三类被测对象的共性字段与扩展位。

本任务需要完成：

- 梳理当前算子层对象表示方式；
- 设计统一的 subject 抽象；
- 约定算子层、模型层、应用层的最小必需信息；
- 为不同层次保留元数据扩展入口；
- 保证后续插件、生成器、执行器可以通过统一入口读取对象信息。

本任务产出后，后续阶段不再为不同层单独发明新的对象主结构。

### G2：统一 MR 表达与生命周期承载能力 ✅

目标：把当前更偏算子层的 MR 记录方式，扩展为可跨层共用的统一关系表示。

本任务需要完成：

- 统一 MR 基本元信息；
- 为来源、验证状态、适用框架、版本信息预留规范位置；
- 支持不同层关系使用不同的 transform / oracle 描述；
- 约定关系的状态流转基础字段；
- 保证现有 repository 查询和导出逻辑可以迁移。

这里不需要把所有细节字段一次定死，但必须让系统具备“可持续扩展”的关系承载能力。

### G3：统一执行记录与证据追踪结构 ✅

目标：让一次测试执行结果在任意层次都能被完整追踪。

本任务需要完成：

- 统一 run 级记录结构；
- 补齐框架版本、随机种子、环境摘要、输入来源等信息；
- 约定结果摘要与失败明细的对应关系；
- 将证据包与测试运行记录建立稳定关联；
- 为实验导出和图表统计保留统一索引入口。

该任务的重点不是新增分析逻辑，而是保证后续数据可以回放、可比、可聚合。

### G4：统一 repository 与 results storage 迁移方案 ✅

目标：在不破坏现有可用数据的前提下完成底层迁移。

本任务需要完成：

- 梳理现有 MRRepository、ResultsManager、证据与 cross results 的结构差异；
- 给出兼容迁移方案；
- 编写数据迁移脚本或兼容读写层；
- 保证旧数据最少可读；
- 为后续新阶段新增数据类型预留统一入口。

该任务应优先保证兼容性和稳定性，而不是一次性追求结构最优。

### G5：统一注册与调度入口整理 ✅

目标：让 catalog / mr / test / report 四类入口在统一底座上继续工作。

本任务需要完成：

- 梳理现有 CLI 入口与数据对象依赖；
- 建立 subject 注册机制；
- 整理不同层的对象发现与加载方式；
- 调整执行流程中的对象分发逻辑；
- 保持现有算子层 CLI 命令可继续运行。

### G6：回归测试与兼容性加固 ✅

目标：在底座重构后，确保现有算子层主链不退化。

本任务需要完成：

- 为统一对象抽象补齐单元测试；
- 为 repository 迁移补齐兼容测试；
- 重点验证现有 `mr generate`、`test batch`、`test report`、`test experiment` 等关键命令；
- 检查 Web 仪表盘依赖的数据接口是否保持可用；
- 形成阶段回归清单。

## 5. 本阶段建议新增或调整的模块

本阶段建议优先新增或调整以下区域：

- `deepmt/core/ir/`
- `deepmt/core/subject_registry.py`
- `deepmt/core/run_manifest.py`
- `deepmt/core/storage/`
- `deepmt/migrations/`
- `tests/unit/test_ir_*.py`
- `tests/unit/test_storage_*.py`

目录命名不要求完全一致，编码智能体可根据现有结构自主微调，但应保持职责集中。

---

## 6. 可运行成果

本阶段结束后，系统至少应能做到：

1. 现有算子层 MR 与测试结果可在新结构下继续读写；
2. 系统可以注册多种层次的被测对象，即使其中部分尚未完全启用；
3. 单次运行结果可以统一追踪到对象、MR、框架、版本、证据；
4. 现有 CLI 主链不回退。

---

## 7. 验收标准

本阶段验收以“底座可用、不破坏主链”为原则：

- 旧有算子层主链功能保持可运行；
- repository 与 results storage 支持新旧结构兼容；
- 至少有一组新结构下的对象注册与运行示例；
- 新增测试覆盖统一对象、统一关系、统一运行记录与迁移逻辑；
- 文档说明清楚后续阶段如何复用该底座。

## 9. 开发完成标志

当以下条件同时满足时，可认为 Phase G 完成：

- 新 IR / subject / run 底座已落地；
- 旧算子层流程在新底座上无明显退化；
- 后续模型层、应用层、第二框架适配都可以直接在此基础上接入；
- 形成阶段总结文档，说明迁移点、兼容点与后续扩展位。
