# 08_Phase_H_第二框架落地与真实跨框架适配.md

## 1. 阶段目标

本阶段目标是把当前“PyTorch + NumPy 参考后端”的一致性测试雏形，升级为“主流深度学习框架之间的真实跨框架测试能力”。

阶段完成后，DeepMT 应至少具备以下能力：

- 除 PyTorch 外，接入一个真实的第二深度学习框架；
- 在一组重点算子上形成双框架可执行闭环；
- 输出可持久化的跨框架一致性结果；
- 将差异结果纳入统一分析与实验视图。

---

## 2. 本阶段要解决的问题

当前系统虽然已经具备跨框架一致性测试模块，但核心仍是 PyTorch 与 NumPy 参考后端的比对。这对于工程验证有价值，但不足以支撑论文中“主流框架系统性比较”的目标。

因此，本阶段的重点不是继续扩充参考后端，而是先把一个真实第二框架做透，使跨框架能力从“研究雏形”升级为“论文可用能力”。

---

## 3. 阶段边界

本阶段建议采用“一个第二框架先做透”的策略，不建议同时浅层接入多个新框架。

建议优先路径：

- 方案一：TensorFlow 作为第二框架；
- 方案二：PaddlePaddle 作为第二框架。

编码智能体可根据现有环境、依赖可获取性和适配难度自主选择，但必须坚持同一阶段只优先打透一个第二框架。

本阶段不做的内容包括：

- 不同时完成 TensorFlow 和 PaddlePaddle 的完整覆盖；
- 不追求一次支持全部算子；
- 不在本阶段处理复杂模型层跨框架训练细节；
- 不为 UI 增加大量新交互页面。

---

## 4. 任务拆分

### H1：第二框架选型与适配边界确认 ✅

目标：明确本阶段要接入的第二框架，并整理其与现有插件抽象之间的适配边界。

本任务需要完成：

- 调研目标框架的安装、运行和 API 风格；
- 对照现有 `FrameworkPlugin` 抽象梳理能力缺口；
- 明确本阶段优先支持的执行模式；
- 确定第一批算子支持范围；
- 输出第二框架适配的边界说明。

### H2：插件抽象收敛与能力补齐 ✅

目标：在尽量不破坏现有 PyTorch 插件的前提下，让插件抽象真正适合多深度学习框架复用。

本任务需要完成：

- 梳理插件公共接口；
- 统一张量构造、dtype 处理、异常包装、容差判断等基础行为；
- 为不同行为差异预留扩展点；
- 将现有 PyTorch 插件整理为更规范的参考实现；
- 明确哪些接口是第二框架必须实现的最小集合。

### H3：第二框架插件第一版实现 ✅

目标：完成第二框架插件的最小可运行实现。

本任务需要完成：

- 新增第二框架插件模块；
- 实现基础算子调用能力；
- 支持输入包装、执行、结果解包和异常规整；
- 与现有 batch / cross 流程打通；
- 保证若框架不可用，系统能给出清晰失败信息。

### H4：算子映射表与重点算子闭环 ✅

目标：建立跨框架可比较的第一批重点算子集合。

本任务需要完成：

- 建立 operator mapping 配置；
- 选择一批高频基础算子作为首批覆盖对象；
- 处理命名差异、参数差异和行为差异；
- 明确“可比较”“需转换”“暂不支持”三类状态；
- 保证至少一批代表性算子能稳定双框架执行。

### H5：跨框架差异分类与结果标准化 ✅

目标：让跨框架测试结果不只是通过/失败，而是能输出有分析价值的差异类型。

本任务需要完成：

- 整理常见跨框架差异类型；
- 在结果记录中加入差异分类；
- 统一 shape / dtype / 数值误差 / 异常行为等差异描述；
- 为后续实验统计提供聚合字段；
- 让报告和仪表盘可以读取这些分类结果。

### H6：跨框架测试命令升级与持久化结果接入 ✅

目标：让 `deepmt test cross` 或相关入口正式支持第二框架闭环。

本任务需要完成：

- 升级现有 cross tester；
- 支持选择第二框架作为真实对比对象；
- 持久化会话级结果；
- 支持读取历史会话进行比较；
- 输出可用于实验章节的基础统计结果。

### H7：第二框架回归测试与环境加固 ✅

目标：确保新框架接入不会破坏当前系统可用性。

本任务需要完成：

- 为第二框架插件补齐单元测试；
- 为 cross tester 补齐双框架集成测试；
- 梳理依赖安装与环境开关；
- 提供最小运行示例；
- 更新健康检查或诊断信息。

---

## 5. 本阶段建议新增或调整的模块

本阶段建议优先新增或调整以下区域：

- `deepmt/plugins/<second_framework>_plugin.py`
- `deepmt/configs/operator_mapping/`
- `deepmt/analysis/cross_framework_tester.py`
- `deepmt/core/framework_registry.py`
- `tests/unit/test_<second_framework>_plugin.py`
- `tests/integration/test_cross_framework_*.py`

---

## 6. 可运行成果

本阶段结束后，系统至少应能做到：

1. 通过统一命令运行 PyTorch 与第二框架的一致性测试；
2. 至少一批重点算子能在两个深度学习框架之间完成可比较执行；
3. 差异结果能按类型落库和聚合；
4. 实验组织器可以读取跨框架结果。

---

## 7. 验收标准

本阶段验收重点看“真实第二框架是否可稳定进入主链”：

- 第二框架插件可独立初始化并执行一批基础算子；
- 双框架 cross test 有稳定结果输出；
- 有一组清晰的算子映射配置和支持状态说明；
- 新增测试覆盖插件能力、差异分类与 cross 执行主链；
- 文档中明确第二框架的支持边界与已知限制。

## 9. 开发完成标志

当以下条件同时满足时，可认为 Phase H 完成：

- 第二框架插件已进入主链；
- 跨框架测试不再局限于 PyTorch vs NumPy；
- 有一批双框架可比较算子；
- 差异分类结果可用于实验统计和案例分析。
