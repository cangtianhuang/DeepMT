# 11_Phase_K_全层MR质量保障与统一知识库治理.md

## 1. 阶段目标

本阶段目标是把当前围绕算子层的 MR 管理方式，升级为覆盖算子层、模型层、应用层的统一治理体系。

阶段完成后，DeepMT 应至少具备以下能力：

- 不同层次 MR 使用统一生命周期管理；
- 关系的来源、验证级别和版本变化可追踪；
- repository 支持跨层查询、筛选和审计；
- 低质量或已废弃关系不会混入默认测试主链；
- 仪表盘和报告可以读取统一质量视图。

---

## 2. 本阶段要解决的问题

随着模型层和应用层接入，MR 的来源将更加多样，质量也会更加不均。若没有统一治理机制，系统很快会出现以下问题：

- 关系是否可信难以判断；
- 不同层关系无法统一比较；
- 旧关系、重复关系、低质量关系会污染测试主链；
- 实验结果很难解释“这些关系为什么可以被纳入统计”。

因此，本阶段的重点是从“有 repository”升级为“有治理能力的知识库”。

---

## 3. 阶段边界

本阶段关注的是治理机制，而不是新增新的复杂生成算法。

本阶段不做的内容包括：

- 不额外扩充大量新 MR 类型；
- 不在本阶段追求复杂在线编辑系统；
- 不把治理做成重型平台；
- 不改变前面阶段已形成的核心生成逻辑。

本阶段强调的是：**统一状态、统一质量口径、统一审计。**

---

## 4. 任务拆分

### K1：MR 生命周期定义与状态流整理

目标：给不同来源、不同层次的关系建立统一状态流。

本任务需要完成：

- 梳理当前关系从生成到使用的实际流程；
- 定义统一生命周期阶段；
- 明确关系何时可进入默认测试集；
- 明确关系何时应被停用或归档；
- 让不同阶段的状态在 repository 中可查询。

### K2：统一质量级别与验证口径

目标：给所有关系建立统一的质量表达方式。

本任务需要完成：

- 整理现有 pre-check、证明、人工确认等验证方式；
- 定义统一的验证级别或质量标签；
- 允许不同层采用不同验证手段，但输出统一质量口径；
- 支持质量等级筛选与统计；
- 避免低质量候选直接进入默认实验。

### K3：关系 provenance 与版本追踪

目标：让每条关系都能追踪其来源和演化过程。

本任务需要完成：

- 为模板、规则、LLM、人工修订等不同来源保留标记；
- 记录 prompt 版本、规则版本或文档来源摘要；
- 支持关系更新与历史留痕；
- 支持比较不同版本关系的变化；
- 为论文中“方法来源与可信性”提供依据。

### K4：统一入库规则与淘汰规则

目标：让 repository 具备“纳入什么、淘汰什么”的明确口径。

本任务需要完成：

- 定义入库前的基本检查；
- 支持重复关系识别和合并策略；
- 明确低质量、失效或不再适用关系的淘汰路径；
- 允许归档但不删除历史痕迹；
- 防止无序增长的候选关系污染知识库。

### K5：仓库审计与治理工具

目标：提供面向开发和实验的治理工具，而不是仅靠手工检查。

本任务需要完成：

- 增加 repository 审计命令或管理入口；
- 支持按层次、来源、质量级别、适用框架筛选；
- 输出仓库质量统计与异常项清单；
- 支持导出待复核列表；
- 让后续论文实验使用的 MR 集合可被明确审计。

### K6：统一质量视图与展示接入

目标：让治理结果不仅存在于底层存储，还能被报告和仪表盘读取。

本任务需要完成：

- 为报告模块增加质量维度统计；
- 为仪表盘增加跨层 MR 状态视图；
- 支持展示来源分布、验证级别分布和失效关系统计；
- 对接实验组织器的数据读取；
- 形成统一的关系质量可视化入口。

### K7：治理回归测试与文档固化

目标：确保治理机制是长期可执行规则，而不是一次性手工整理。

本任务需要完成：

- 补齐生命周期、版本追踪、审计逻辑的测试；
- 为仓库治理给出使用说明；
- 形成关系质量策略文档；
- 说明不同层关系在什么条件下可纳入论文实验。

## 5. 本阶段建议新增或调整的模块

本阶段建议优先新增或调整以下区域：

- `deepmt/mr_governance/`
- `deepmt/mr_generator/base/mr_repository.py`
- `deepmt/commands/repo_*.py`
- `deepmt/analysis/repo_audit.py`
- `deepmt/ui/routers/api.py`（质量统计扩展）
- `tests/unit/test_repo_governance.py`

---

## 6. 可运行成果

本阶段结束后，系统至少应能做到：

1. 三层 MR 进入同一知识库治理体系；
2. 每条 MR 可追踪来源、质量级别和状态；
3. 仓库支持按质量和适用范围筛选；
4. 低质量关系不会默认进入测试主链；
5. 仪表盘和报告可读取统一质量视图。

---

## 7. 验收标准

本阶段验收重点是“知识库是否从存储工具升级为治理系统”：

- 生命周期和质量口径在不同层间统一；
- 仓库审计工具可输出可读结果；
- 关系版本变化和来源信息可追踪；
- 实验可明确指定纳入的 MR 质量范围；
- 新增测试覆盖治理主逻辑。

---

## 8. 与论文章节的对应关系

本阶段主要支撑以下论文内容：

- 第 3.5 节：多阶段质量保障与统一入库机制；
- 第 4.2.4 节：关系知识库存储结构；
- 第 5.2 节：MR 生成质量评估的数据基础。

---

## 9. 开发完成标志

当以下条件同时满足时，可认为 Phase K 完成：

- 三层关系已进入统一治理框架；
- repository 具备生命周期、质量和审计能力；
- 报告与实验可以基于质量口径稳定筛选关系；
- 仓库管理从“查数据”升级为“管质量”。
