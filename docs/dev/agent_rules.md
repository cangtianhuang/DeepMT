# 编码智能体执行规范与开发原则

> 本文档由 `06_编码智能体执行规范.md` 与 `00_开发总览与执行原则.md`（推进原则节）合并而成，用于约束编码智能体的工作方式，确保开发节奏稳定、任务边界清晰、产出能够直接并入主链。

---

## 1. 项目推进总原则

### 原则一：先研究主链，后扩展功能面

当前后续工作的优先级固定为：

1. 算子数据层；
2. MR 批量生成；
3. 批量测试执行；
4. 结果报告与缺陷证据；
5. 演示与工程加固。

任何不直接服务这条主链的工作，都应降低优先级。

### 原则二：先 PyTorch 闭环，后多框架扩展

- PyTorch 作为第一落地框架；
- TensorFlow / PaddlePaddle 先完成接口预留与适配边界设计；
- 当 PyTorch 跑通批量测试与结果分析后，再扩展第二框架做一致性实验。

### 原则三：每个阶段都要产出"可运行成果"

每个阶段结束后必须交付一个立即可验证的成果：可查询的目录、可批量生成的知识库、可批量运行的命令、可直接阅读的报告、可复现的缺陷案例包。

### 原则四：优先保证证据链，而不是追求花哨界面

实验记录、失败样本、参数配置、MR 版本、框架版本这些证据必须前置建设。没有证据链，无法支撑论文结论，也无法让真实缺陷被确认。

## 2. 工作行为准则

### 2.1 一次只解决一个明确问题

建议把任务控制在"单次开发可以独立完成并验证"的粒度，例如：

- 完成 `InputGenerator`；
- 完成 `mr batch-generate`；
- 完成结果 JSON 输出；
- 完成失败样本持久化。

不建议把多个阶段任务打包成一次大改动。

### 2.2 先补主链缺口，不做过早优化

遇到选择时，优先实现：

- 能让主链跑通的功能；
- 能提升研究可信度的功能；
- 能减少伪阳性的功能。

暂缓实现：

- 花哨界面；
- 复杂抽象层；
- 低收益的泛化设计；
- 与论文主线无关的辅助工具。

## 3. 每类任务建议写法

### 3.1 任务描述

任务描述尽量包含三部分：为什么做、改哪里、产出是什么。

例如：

> 为批量测试提供自动输入能力，在 `core/input_generator.py` 中实现基于 `input_specs` 的输入生成，产出可被 `test batch` 直接调用的接口。

### 3.2 验收标准

验收标准保持简短，只写最核心的"完成标志"：

- 命令可运行；
- 输出结构正确；
- 关键单元测试通过；
- 可被下一阶段直接使用。

### 3.3 输出要求

每个任务完成后，固定输出：

- 修改了哪些文件；
- 做成了什么；
- 怎么验证；
- 下一步最自然的衔接任务。

---

## 4. 测试与验证原则

### 4.1 优先补最靠近主链的测试

建议优先覆盖：

- 数据结构解析；
- 目录与知识库读写；
- 输入生成；
- 批量命令调度；
- 结果输出格式。

### 4.2 外部依赖与本地测试分离

涉及网络、LLM、真实框架大运行的测试，应与纯单元测试分开，避免影响基本开发效率。

### 4.3 失败要可解释

当任务没有完全做成时，输出应明确说明：

- 哪一步完成；
- 哪一步未完成；
- 当前阻塞点是什么；
- 推荐下一步如何继续。

---

## 5. 文档同步原则

每完成一个阶段中的关键任务，应同步更新 `docs/dev/status.md` 及对应阶段文档中的完成情况。同步方式应简洁，重点更新：

- 已完成项；
- 未完成项；
- 下一步优先任务。

---

## 6. 对"可投入生产使用"的理解

本项目的生产化目标是"作为深度学习框架测试工具持续运行"，而不是在论文周期内做成重型平台。

正确方向是：

- 让命令稳定；
- 让结果可追踪；
- 让扩展有边界；
- 让环境可复现。

而不是过早投入：

- 大规模平台前端；
- 复杂权限系统；
- 分布式服务治理。

---

## 7. 最重要的一条

所有开发都应服务于以下最终展示：

> 选定一组算子，系统自动生成蜕变关系，自动执行测试，自动输出可疑问题和报告，并能拿出至少一个可复现的真实案例。

只要某项开发不能让这条展示链更顺畅，它的优先级就应下降。
