## 1. 阶段定位

本阶段是整个系统的地基阶段。其核心任务不是“多抓一些 API”，而是建立一套**能真正被 MR 生成器和测试执行器稳定消费**的算子数据层。

旧版 Phase A 更像是字段扩展说明；重写后，本阶段要解决的是：

- 什么算子值得测；
- 算子如何被统一描述；
- 如何自动拿到足够可靠的输入约束；
- 如何让目录数据在后续阶段可直接用于生成和测试。

一句话概括：

> Phase A 的产物应是一份可信的、可批量维护的、可直接驱动测试的算子目录。

---

## 2. 本阶段目标

### 目标一：建立统一算子目录模型

算子目录必须同时服务三类消费者：

- MR 生成器；
- 测试执行器；
- 报告与实验统计模块。

因此目录应关注“测试需要什么”，而不是只保存文档元信息。

### 目标二：建立输入约束的可用表示

输入约束是自动化测试能否跑通的关键。目录中对 `dtype`、`shape`、`value_range` 的描述，不求一开始做到完美，但必须足以支撑批量生成**大多数合法样本**。

### 目标三：形成框架化的数据采集流程

PyTorch 先行实现，但目录采集、文档 URL、HTML 解析、自动丰富等逻辑都必须保留框架参数化能力，避免后续扩展时重写主链。

### 目标四：定义重点算子集

不是所有 API 都应同等优先级处理。必须明确第一批重点算子集合，用于：

- 打通开发主链；
- 形成论文实验基线；
- 进行演示；
- 提高真实缺陷发现概率。

建议首批覆盖以下类别：

- activation
- elementwise math
- reduction
- linear algebra
- normalization
- loss

## 5. 开发任务

## A1. 固化目录 Schema，停止字段层反复变动

### 目标

把当前 YAML 与 IR 字段定义视为第一版正式协议，后续除非明显阻塞，不再频繁改字段。

### 任务说明

- 检查 `OperatorEntry`、`OperatorIR`、YAML、CLI 输出字段是否完全一致；
- 把“哪些字段用于执行，哪些字段用于展示，哪些字段用于未来扩展”写清楚；
- 对 `api_style`、`api_path`、`module_class` 的语义做最终确认；
- 对 `input_specs_auto` 的语义做固定：它表示“当前仍需人工抽检”，而不是“不可用”。

### 完成标准

- Schema 在目录、IR、CLI 中保持一致；
- 后续阶段无需再因字段意义不清而返工。

---

## A2. 建立“重点算子集”而不是平均覆盖

### 目标

为后续 MR 生成、批量测试、实验评估提供明确的第一批测试对象。

### 任务说明

从已抓取的 PyTorch API 中筛出一批高价值算子，形成 `seed operator set`。筛选原则：

- 文档清晰，输入约束相对明确；
- 数学性质丰富，适合 MR 生成；
- 覆盖不同类别；
- 包含常用、基础、容易暴露数值问题的算子。

建议第一批数量控制在 **30~80 个**，不要过大。

### 完成标准

- 有一份明确的重点算子列表；
- 列表可直接作为 `mr batch-generate` 和 `test batch` 的默认输入来源。

---

## A3. 让 `input_specs` 面向“自动测试”而不是“文档描述”

### 目标

将 `input_specs` 从说明性字段提升为可执行输入约束。

### 任务说明

重点关注以下问题：

- `dtype` 是否足以驱动随机样本生成；
- `shape` 是否能被程序解析；
- `value_range` 是否能避免大量无意义异常；
- 多参数算子能否表达参数间基本依赖。

当前阶段不追求复杂约束求解，只要求表达以下最常见约束：

- 单输入 tensor 约束；
- 固定维度或最小维度；
- 有界值域；
- 少量离散枚举参数；
- 必选与可选参数区分。

### 完成标准

- 第一批重点算子的 `input_specs` 能被后续输入生成器稳定消费；
- 对大多数算子，不再依赖人工逐个构造输入。

---

## A4. 建立 enrich 流程的质量分层

### 目标

不要把自动 enrich 结果简单视为“已完成”，而是建立轻量质量层次，方便后续使用时做不同处理。

### 任务说明

建议把目录条目分为三类状态：

- **confirmed**：人工确认过，可直接用于批量测试；
- **auto-usable**：自动生成，质量可接受，可进入主链但应允许失败回收；
- **weak**：信息不足，仅用于展示或后续补齐。

该状态可以通过单独字段或复用现有标记实现，不要求复杂。

### 完成标准

- 后续阶段能区分“适合直接批量跑”的算子与“仅作候选”的算子；
- 失败统计时能避免把弱数据带来的错误全部误判为框架缺陷。

---

## A5. 目录同步流程工程化

### 目标

把目录更新变成一条可复用的操作链，而不是临时脚本行为。

### 任务说明

需要明确一条推荐流程：

1. 更新 API 列表；
2. 导入或增量合并目录；
3. enrich 输入约束；
4. 对重点算子做抽检；
5. 输出本次目录变更摘要。

同时保留以下能力：

- 全量重建；
- 增量更新；
- 单算子重跑 enrich；
- 导出重点算子集。

### 完成标准

- 目录维护方式清晰；
- 新版本 PyTorch 到来后，可以快速完成一次目录更新。

---

## A6. 为跨框架扩展保留边界，但不展开实现

### 目标

确保本阶段数据层设计不会把后续 TensorFlow / PaddlePaddle 扩展堵死。

### 任务说明

- 所有目录访问接口保留 `framework` 参数；
- 文档 URL 模板、API 列表来源、HTML 主体定位维持框架可配置；
- 不在 Phase A 内实现 TF / Paddle 的完整采集链路；
- 只要求确保新增框架时不需要推翻 PyTorch 路线。

### 完成标准

- Phase A 的数据模型与采集接口对新框架是可复用的；
- 多框架扩展成为后续新增适配工作，而不是架构重构工作。

---

## 6. 本阶段建议实现顺序

建议按以下顺序推进：

1. 固化 Schema；
2. 定义重点算子集；
3. 清理并补齐重点算子的 `input_specs`；
4. 加入轻量质量状态；
5. 固化目录同步流程；
6. 保持跨框架接口不变。

其中最关键的是 **重点算子集 + 可执行 input_specs**。这两项做不好，后续批量生成与批量测试都会陷入大量低质量失败。

---

## 7. 本阶段完成后的直接收益

如果 Phase A 完成得好，后续将立即获得三项收益：

1. `InputGenerator` 可以快速落地；
2. `mr batch-generate` 可以直接面向算子集合运行；
3. 失败案例更可能指向真实框架问题，而不是脏数据问题。

因此，Phase A 虽然是数据层，但它实际上决定了整个项目后续研究结果的可信度。
