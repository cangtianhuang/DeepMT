# Phase C：测试执行与跨框架适配

## 1. 阶段定位

Phase C 是整个项目从“会生成”走向“能发现问题”的分水岭。

前两个阶段解决的是“测试输入从哪里来”的问题；本阶段解决的是“如何把这些输入真正转化为稳定、批量、可复现的测试执行结果”。

一句话概括：

> Phase C 的目标是打通可批量运行的测试主链，并为后续跨框架一致性测试建立统一执行边界。

---

## 2. 本阶段目标

### 目标一：建立批量测试执行主链

系统需要从算子目录和 MR 知识库中读取数据，自动生成输入，调用框架插件执行测试，并把结果保存下来。

### 目标二：规范插件调用边界

目前项目已经有 PyTorch 相关能力，但接口仍偏向原型。需要把插件能力收束成统一的执行协议，以支撑多框架扩展。

### 目标三：让测试结果可复现、可回放

测试执行不是即时打印完成就结束。每一次失败都应该能够追溯到：

- 哪个算子；
- 哪条 MR；
- 哪组输入；
- 哪个框架版本；
- 哪种异常或违例。

---

## 3. 当前基础

当前已具备：

- `deepmt test operator` 单算子测试入口；
- PyTorch 数据层基础信息；
- MR 知识库存储；
- CLI 体系可承载新命令。

但仍缺少三项关键能力：

- 可自动生成大量合法输入；
- 可批量执行 MR 集合；
- 可结构化保存测试结果。

---

## 4. 本阶段交付物

本阶段结束后，应至少交付：

1. `InputGenerator`；
2. 统一插件 `invoke` 接口；
3. 批量测试命令；
4. 结果存储结构；
5. 可用于后续复现和分析的失败样本记录。

---

## 5. 开发任务

## C1. 输入生成器落地

### 目标

让系统不再依赖人工手写样本，而能根据 `input_specs` 自动生成合法输入。

### 任务说明

输入生成器只需解决最常见场景，不追求一次覆盖全部复杂约束。第一版应重点支持：

- 张量 dtype 随机选择；
- 维度与形状生成；
- 有界值域；
- 少量边界值注入；
- 必填参数自动填充。

建议采用“随机样本 + 少量边界样本”的混合策略，这比纯随机更有利于发现数值问题。

### 完成标准

- 重点算子的 `input_specs` 可被自动转成一批可执行输入；
- 无需人工为每个算子单独写输入构造函数。

---

## C2. 插件执行接口规范化

### 目标

统一不同框架算子的调用方式，让测试引擎只关心“执行算子”，而不关心具体框架实现细节。

### 任务说明

插件基类应围绕一个简单能力展开：

- 给定 `OperatorIR` 与输入字典，执行算子并返回输出。

PyTorch 插件在本阶段要完整实现，重点处理：

- `function` 风格调用；
- 后续兼容 `module` 风格；
- 框架对象与原生 Python 值的必要转换；
- 异常信息的结构化返回。

当前不要求同时实现 TensorFlow / PaddlePaddle，只要求接口足够清晰。

### 完成标准

- 测试引擎能只依赖 `invoke` 而不写框架分支逻辑；
- PyTorch 插件可稳定服务批量测试。

---

## C3. 批量测试执行器

### 目标

实现从 MR 知识库到批量执行再到结果记录的标准流程。

### 任务说明

建议测试执行器围绕一条固定主链实现：

1. 选取算子与 MR；
2. 读取算子输入约束；
3. 自动生成输入样本；
4. 执行原始测试；
5. 执行变换后测试；
6. 验证 oracle；
7. 记录通过、失败、异常和样本信息；
8. 汇总为可读统计。

批量入口应支持按以下维度筛选：

- framework
- operator
- category
- mr_id
- n_samples

### 完成标准

- 系统能对一批算子、一批 MR 连续执行测试；
- 执行结果可以结构化保存，而不是只在终端打印。

---

## C4. 结果存储与失败样本沉淀

### 目标

让测试结果成为后续研究和复现的基础资产。

### 任务说明

建议结果层至少记录以下信息：

- 执行时间；
- 框架与版本；
- 算子与 MR 标识；
- 输入样本摘要；
- 原始输出与变换后输出摘要；
- oracle 判定结果；
- 异常类型；
- 可选的失败样本持久化路径。

当前阶段不要求复杂数据库设计，JSON + 目录化存储也可以，只要可读、可查、可复现。

### 完成标准

- 对任何失败结果，都能回溯其来源；
- Phase D 可以直接基于这些结果做统计与分析。

---

## C5. 为跨框架一致性测试预留统一执行面

### 目标

虽然当前主力是 PyTorch，但本阶段必须把“同一 MR 在不同框架上执行”作为未来扩展目标嵌入设计。

### 任务说明

应提前统一以下概念：

- `OperatorIR` 的框架无关表达；
- 插件输入与输出的统一包装；
- 数值比较容忍度的配置入口；
- 同一算子跨框架映射表的挂载位置。

本阶段只要求接口预留，不要求做全量跨框架测试。

### 完成标准

- Phase C 完成后，引入第二框架时主要是补适配器与映射表，而不是重写执行器。

---

## 6. 本阶段建议实现顺序

建议实现顺序：

1. `InputGenerator`；
2. `invoke` 统一接口；
3. `test batch` 命令；
4. 结果存储；
5. 少量失败复现工具。

其中最关键的是 **输入生成器 + 批量测试执行器**。如果没有这两项，DeepMT 仍停留在“半自动研究原型”阶段。

---

## 7. 本阶段完成后的意义

本阶段完成后，项目会第一次具备“真实发现问题”的基础能力。

这意味着：

- 研究上可以开始统计缺陷检测结果；
- 演示上可以展示自动测试闭环；
- 工程上可以持续运行并积累失败样本。

因此，Phase C 实际上是整个项目最关键的落地阶段。
