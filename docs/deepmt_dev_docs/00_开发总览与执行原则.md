# 开发总览与执行原则

## 1. 项目总目标

DeepMT 后续开发只围绕一个核心问题展开：

> 如何把“面向深度学习框架的分层蜕变测试体系研究”落成一个既能支撑硕士论文研究结论、又能实际发现框架问题、还能被演示和持续运行的软件系统。

为此，系统最终必须同时具备三种能力：

### 1.1 研究闭环能力

系统需要支持以下研究动作：

- 自动生成蜕变关系；
- 在多个算子上批量执行测试；
- 通过控制实验和真实框架测试得到可分析结果；
- 输出可复现、可统计、可对比的实验数据；
- 支撑论文中的 RQ1-RQ4。

### 1.2 演示交付能力

系统需要能以较短路径展示完整价值链：

- 从框架文档和算子目录启动；
- 自动生成 MR；
- 自动执行批量测试；
- 自动生成缺陷线索和可读报告；
- 在演示环境中稳定复现至少一类真实问题或高价值异常。

### 1.3 生产落地能力

系统不需要一步做到企业级平台，但必须具备基础生产属性：

- 模块边界清晰；
- 核心流程可重复执行；
- 结果可追踪、可回放；
- 框架适配可扩展；
- 失败有日志，结果有落库，配置可管理。

---

## 2. 总体推进原则

### 原则一：先研究主链，后扩展功能面

当前阶段不追求同时完成三层体系的全部细节。必须先把**算子层主链**做实，因为：

- 算子层最容易形成真实缺陷发现；
- 算子层最适合作为跨框架一致性入口；
- 算子层结果最容易形成论文中的定量实验；
- 当前已有代码基础主要集中在算子层。

因此，近期主线是：

**算子目录 → MR 生成 → 批量测试 → 结果分析 → 缺陷证据包**

模型层和应用层先保留接口与后续扩展位，不在当前阶段分散主力开发资源。

### 原则二：先 PyTorch 闭环，后多框架扩展

多框架是论文的重要组成，但不应在没有单框架稳定主链的情况下同时展开。当前建议：

- **PyTorch 作为第一落地框架**；
- TensorFlow / PaddlePaddle 先完成接口预留与适配边界设计；
- 当 PyTorch 跑通批量测试与结果分析后，再扩展第二框架用于一致性实验。

### 原则三：每个阶段都要产出“可运行成果”

文档不是任务罗列，而是要求每个阶段结束后都能交付一个立即可验证的成果，例如：

- 可查询的算子目录；
- 可批量生成的 MR 知识库；
- 可批量运行的测试命令；
- 可直接阅读的报告；
- 可复现的缺陷案例包。

### 原则四：优先保证证据链，而不是追求花哨界面

展示系统可以后置，但实验记录、失败样本、参数配置、MR 版本、框架版本这些证据必须前置建设。没有证据链，无法支撑论文结论，也无法让真实缺陷被确认。

---

## 3. 阶段规划

## Phase A：算子数据层完善

目标：形成可信、可扩展、可自动维护的算子目录与输入约束数据层，为 MR 生成和批量测试提供可靠输入。

## Phase B：算子层 MR 生成与知识库

目标：把当前单算子流水线升级为可批量运行、可治理、可追踪质量的 MR 生成系统。

## Phase C：测试执行与跨框架适配

目标：打通从知识库到批量执行再到结果存储的运行主链，并规范插件接口。

## Phase D：缺陷分析、实验闭环与研究结论

目标：让系统能稳定产出支持论文实验章节的数据、图表来源和缺陷证据。

## Phase E：演示交付与生产化加固

目标：补齐演示路径、工程加固、部署方式与运维基础，使系统可用于展示与持续运行。

---

## 4. 当前基础与主要缺口

## 已有基础

项目当前已经具备：

- 单算子 MR 生成器；
- 随机预筛与 SymPy 证明能力；
- MR 知识库存储；
- PyTorch 算子目录和文档抓取工具；
- 基础 CLI 体系；
- 单元测试骨架。

## 主要缺口

当前距离论文闭环还差四类关键能力：

### 4.1 数据层仍缺少“测试可用性保证”

算子目录已经存在，但并不等价于“测试就能稳定运行”。当前更需要关注：

- 输入约束是否足够支撑自动生成合法样本；
- 文档与目录是否能稳定同步；
- 重点算子集是否被优先覆盖。

### 4.2 生成层仍缺少“批量与治理”

当前更像研究原型，而不是可长期运行的生成系统。需要补齐：

- 批量生成；
- 重跑与跳过策略；
- MR 生命周期管理；
- 质量标签与版本追踪。

### 4.3 执行层仍缺少“面向研究的测试主链”

论文需要的是“批量缺陷发现能力”，因此必须构建：

- 自动输入生成；
- 批量测试执行器；
- 标准化插件调用接口；
- 结果持久化与统计汇总。

### 4.4 结果层仍缺少“结论支撑”

只有原始测试结果还不够，还需要：

- 失败样本沉淀；
- 可复现缺陷记录；
- 去重与归类；
- 与 RQ1-RQ4 对应的实验数据视图。

---

## 5. 近期优先级

近期优先级只保留以下一条主线：

1. 重写并固化 Phase A 数据层；
2. 完成算子层批量 MR 生成；
3. 完成批量测试执行；
4. 完成结果报告与缺陷证据输出；
5. 在 PyTorch 上形成第一版研究闭环；
6. 再引入第二框架做一致性验证。

不建议当前优先投入以下方向：

- 复杂 Web UI；
- 模型层大规模自动生成；
- 应用层 LLM 复杂智能体编排；
- 过早的性能优化；
- 过宽的框架支持面。

---

## 6. 最终交付形态

当本组文档对应开发完成后，DeepMT 应至少能以如下方式被使用：

1. 选择框架和算子集合；
2. 自动准备算子元数据和输入约束；
3. 自动生成并保存 MR；
4. 自动批量执行测试；
5. 自动输出失败样本、统计摘要和可疑缺陷列表；
6. 支持研究实验与课堂/答辩演示。

这意味着系统至少要有四个稳定入口：

- **catalog**：管理算子与文档；
- **mr**：生成与治理蜕变关系；
- **test**：执行与复现测试；
- **report**：分析与输出结果。

后续所有开发应围绕这四个入口展开，而不是新增分散工具。
