# 面向深度学习框架的蜕变关系自动生成与分层测试体系研究

## 摘要

```latex

\begin{abstract}
	深度学习框架作为人工智能应用的重要基础软件，其可靠性将直接影响上层模型训练、推理过程的质量以及系统的可靠性。然而，深度学习框架测试存在典型的“测试预言机”问题，即在给定输入的情况下，往往缺乏可参照的期望输出用以判定执行结果是否正确，传统基于预期输出的测试方法难以直接适用。蜕变测试作为一种特殊黑盒测试方法，其不依赖执行的预期输出，而是通过验证多次相关执行之间是否满足特定的蜕变关系来检测缺陷，为上述问题提供了可行了解决思路。现有研究大多针对代码系统中的特定组件展开，缺乏系统化的测试体系；同时，蜕变关系的构造仍以人工设计为主，在很大程度上限制了蜕变测试的覆盖范围与规模化应用。针对上述问题，本研究围绕深度学习框架的分层蜕变测试体系与蜕变关系自动生成方法两个方向展开，主要工作如下：

	（1）提出了一种面向深度学习框架的分层蜕变测试方法。对深度学习框架软件栈进行了高度抽象，将测试任务划分为算子层、模型层和应用层三个相互独立的层次，分别定义各层的测试目标、蜕变关系类型与结果验证策略。此外，设计了统一中间表示与跨框架映射机制，将不同框架间的API差异封装于适配层之下，为多框架测试提供统一接口。

	（2）提出了一种多源融合的蜕变关系自动生成方法。针对不同层次采用差异化的生成策略：算子层依据形式化规约与数学性质模板推导蜕变关系；模型层结合神经网络结构静态分析与数据增强策略库生成蜕变关系；应用层借助大语言模型从框架文档中提取领域知识，生成语义层面的蜕变关系。在此基础上，进一步构建了包含随机采样快速筛选与基于符号计算的形式化证明的校验流程，保障蜕变关系生成的质量。

	（3）设计并实现了自动化测试框架原型系统DeepMT。系统采用了微内核与插件化架构，对核心引擎与框架适配的逻辑解耦分离，通过独立的适配器插件实现了对PyTorch、TensorFlow、PaddlePaddle等主流框架的支持。系统集成了蜕变关系生成、测试用例调度、跨框架执行以及缺陷检测分析等全流程功能。

	基于上述方法与系统，本文对多个主流深度学习框架开展了实证研究，从蜕变关系生成质量、缺陷检测能力和跨框架一致性等三个方面进行评估。实验结果表明，本研究提出的方法能够较好地支持深度学习框架的分层蜕变测试，在一定程度上提高了蜕变关系构造的自动化水平。相关工作可为深度学习框架的蜕变测试工具设计提供工程参考，也为蜕变测试在大规模系统软件中的进一步应用积累了实践经验。
\end{abstract}

```

\section{问题建模与总体思路}

\subsection{问题定义与目标}

深度学习框架的复杂性决定了无法用单一的“输入-输出”断言来覆盖其全部行为。为建立分层测试的理论基石，必须首先从形式化角度对框架及其蜕变测试目标进行严谨建模。

\paragraph{定义 3.1（深度学习框架的分层抽象）}
深度学习框架 $F$ 在逻辑上被建模为三元组 $F = (\mathcal{O}, \mathcal{C}, \mathcal{A})$。其中：
\begin{itemize}
    \item $\mathcal{O} = \{o_1, o_2, \ldots, o_n\}$ 为算子级执行单元集合，提供原子数学计算能力；
    \item $\mathcal{C}$ 为模型级计算图或编译执行对象集合，负责算子的拓扑组合与执行期优化；
    \item $\mathcal{A}$ 为应用级任务执行管道或高层工作流集合，面向具体的领域语义任务（如图像分类流水线）。
\end{itemize}

上述抽象层次的解耦直接推导出了本研究的三个测试维度。由于在上述任意层次中，单次执行均难以获得绝对精准的测试预言机，本文引入蜕变测试方法对其进行替代判定。考虑到深度学习浮点计算的特性，验证谓词必须内化容差机制。

\paragraph{定义 3.2（带容差的蜕变关系与违例判定）}
给定目标组件 $P$（$P \in \mathcal{O} \cup \mathcal{C} \cup \mathcal{A}$）及其输入空间 $\mathcal{X}$、输出空间 $\mathcal{Y}$。一条适用于深度学习框架的蜕变关系定义为二元组：
\[
    MR = (T_{in}, R_{out}^{\epsilon})
\]
其中：
\begin{itemize}
    \item $T_{in}: \mathcal{X} \rightarrow \mathcal{X}$ 为输入变换函数，用于从源输入 $x$ 生成变换输入 $x' = T_{in}(x)$；
    \item $R_{out}^{\epsilon}: \mathcal{Y} \times \mathcal{Y} \rightarrow \{0, 1\}$ 为带容差参数 $\epsilon \ge 0$ 的输出验证谓词。它衡量源输出 $y = P(x)$ 与变换输出 $y' = P(x')$ 之间的计算差异是否满足理论预期。
\end{itemize}

当且仅当 $R_{out}^{\epsilon}(P(x), P(T_{in}(x))) = 1$ 时，判定程序在该输入下满足预设验证关系。反之，若触发下述条件：
\begin{equation}
    R_{out}^{\epsilon}(P(x), P(T_{in}(x))) = 0
\end{equation}
则构成一次蜕变关系违例（MR Violation）。该定义为后续各层的模式识别、规则实例化与缺陷判定提供了统一的形式化基础。

\subsection{分层蜕变测试统一方法框架}

虽然算子层、模型层与应用层的抽象级别不同，但本文将它们统一纳入同一执行与质量控制框架中。各层的共同逻辑范式表现为：
\begin{enumerate}
    \item \textbf{共同目标}：各层均面向预言机缺失问题，旨在生成可执行的 MR。
    \item \textbf{统一范式}：各层都遵循“源输入—变换输入—关系验证”的基础测试结构。
    \item \textbf{统一管控}：各层生成的候选 MR 均无差别地进入统一的多阶段质量保障流水线，最终写入全局一致的关系库。
\end{enumerate}

\section{算子层蜕变关系自动生成方法}

算子层是验证框架数值正确性的底层支撑。本节提出一种从算子属性标签到 MR 实例的完整算法链路，通过映射规则库与模板匹配机制实现批量自动化生成。

\subsection{算子签名获取与属性映射}

给定任意算子 $o_i \in \mathcal{O}$，首先通过代码反射与文档解析器提取其形式化签名 $\Sigma_{o_i}$（包含输入张量规范、数据类型与可调参数）。随后，基于预定义的人工规则库或数学规约解析，为算子绑定一个或多个数学属性标签 $\mathcal{L}(o_i) \subseteq \Lambda$。典型标签包括线性齐次性（$\lambda_{linear}$）、交换律（$\lambda_{comm}$）、单调性（$\lambda_{mono}$）等。

在实际生成中，一个算子常常命中多个标签（例如 \texttt{add} 算子同时具有交换律和结合律）。系统会对互斥的属性模板进行约束剪枝，并行生成多条不同类型的 MR 实例。

\subsection{模板驱动的实例化算法}

针对匹配到的 $\lambda$ 标签，系统激活对应的数学性质模板进行实例化。算子层完整的生成链路如算法 \ref{alg:op_mr_gen} 所示。

\begin{algorithm}[htbp]
    \caption{算子层 MR 自动生成算法}
    \label{alg:op_mr_gen}
    \begin{algorithmic}[1]
        \Require 算子集合 $\mathcal{O}$，数学性质模板库 $\mathcal{T}_{math}$
        \Ensure 算子层蜕变关系集合 $\mathcal{M}_{op}$
        \State $\mathcal{M}_{op} \leftarrow \emptyset$
        \For{each $o_i \in \mathcal{O}$}
        \State $\Sigma_{o_i} \leftarrow \text{ExtractSignature}(o_i)$
        \State $\mathcal{L}(o_i) \leftarrow \text{MapPropertyTags}(\Sigma_{o_i}, \text{Docs})$
        \For{each $\lambda \in \mathcal{L}(o_i)$}
        \State $T_{temp} \leftarrow \text{FetchTemplate}(\mathcal{T}_{math}, \lambda)$
        \If{$\text{CheckPreconditions}(T_{temp}, \Sigma_{o_i})$}
        \State $T_{in}, R_{out} \leftarrow \text{InstantiateCode}(T_{temp}, \Sigma_{o_i})$
        \State $\epsilon \leftarrow \text{CalculateAdaptiveTolerance}(\Sigma_{o_i})$
        \State $MR \leftarrow (T_{in}, R_{out}^{\epsilon})$
        \State $\mathcal{M}_{op} \leftarrow \mathcal{M}_{op} \cup \{MR\}$
        \EndIf
        \EndFor
        \EndFor
        \Return $\mathcal{M}_{op}$
    \end{algorithmic}
\end{algorithm}

以 \texttt{matmul} 算子为例：算法解析到其具备对加法的分配律标签 $\lambda_{distrib}$，匹配到模板后，检查前置条件（要求三个输入张量的维度必须满足矩阵乘法与加法广播规则），进而实例化输入变换 $T_{in}(A, B, C) = (A, B+C)$，并输出验证谓词 $R_{out}^{\epsilon}: \|f(A, B+C) - (f(A, B) + f(A, C))\|_\infty \le \epsilon$。

\section{模型层蜕变关系自动生成方法}

模型层不再针对孤立算子，而是面向计算图拓扑。本节提出一种基于图模式匹配与运行态约束校验的 MR 生成方法。

\subsection{子图模式库与约束边界}

方法首先定义了结构化模式库 $\Omega = \{\omega_1, \omega_2, \ldots, \omega_k\}$。一个模式 $\omega$ 不仅包含节点类型与拓扑连接要求，还必须声明严格的运行态约束，以防止关系泛化引发误报。模式定义的核心元素包括：
\begin{itemize}
    \item \textbf{拓扑约束}：如必须存在从 \texttt{Conv2d} 到 \texttt{GlobalAvgPool} 的有向路径。
    \item \textbf{参数约束}：如卷积层的 Padding 策略必须为对称或有效（Valid）填充，且平移幅度在感受野允许像素内。
    \item \textbf{执行态约束}：模型必须处于推理模式（\texttt{Eval Mode}），即强行禁用 \texttt{Dropout} 并冻结 \texttt{BatchNorm} 的统计量更新。
\end{itemize}

\subsection{模式识别与关系实例化算法}

在满足严格约束的前提下，系统对模型计算图进行模式匹配并实例化拓扑扰动（语义保持的等价图变换）或特征不变性规则。完整逻辑见算法 \ref{alg:model_mr_gen}。

\begin{algorithm}[htbp]
    \caption{模型层模式识别与关系实例化算法}
    \label{alg:model_mr_gen}
    \begin{algorithmic}[1]
        \Require 模型计算图 $G_M = (V, E)$，子图模式库 $\Omega$
        \Ensure 模型层蜕变关系集合 $\mathcal{M}_{model}$
        \State $\mathcal{M}_{model} \leftarrow \emptyset$
        \State $L_{topo} \leftarrow \text{TopologicalSort}(G_M)$
        \For{each $\omega \in \Omega$}
        \If{$\text{SubgraphMatch}(L_{topo}, \omega.\text{topology})$}
        \If{$\text{VerifyConstraints}(G_M, \omega.\text{constraints})$}
        \Comment{检查 Eval 模式、随机性等}
        \State $T_{in} \leftarrow \text{BindPerturbation}(\omega.\text{transform})$
        \State $R_{out}^{\epsilon} \leftarrow \text{BindAssertion}(\omega.\text{assertion})$
        \State $\mathcal{M}_{model} \leftarrow \mathcal{M}_{model} \cup \{(T_{in}, R_{out}^{\epsilon})\}$
        \EndIf
        \EndIf
        \EndFor
        \Return $\mathcal{M}_{model}$
    \end{algorithmic}
\end{algorithm}

例如，针对“确定性推理流”模式，当算法遍历 $G_M$ 确认无任何随机采样算子且处于 \texttt{Eval} 模式时，将实例化恒等输入变换 $T_{id}$，并要求输出具有严格的一致性（$\epsilon = 0$）。

\section{应用层蜕变关系自动生成方法}

应用层面向高层任务。本节建立了一条从知识检索、LLM 结构化抽取到代码规范化校验的完整闭环链路。

\subsection{领域知识检索与结构化上下文构建}

为了解决 LLM 孤立生成的幻觉问题，首先基于 RAG 技术构建检索上下文。步骤如下：
\begin{enumerate}
    \item \textbf{意图确定}：解析目标管道任务 $A$（如文本情感分类）。
    \item \textbf{知识检索}：从预置的框架官方文档、任务规范与接口说明库 $\mathcal{K}$ 中进行稠密检索。
    \item \textbf{去噪过滤}：利用 TF-IDF 等机制剔除低相关代码片段，保留高相关的输入输出规约。
    \item \textbf{提示组装}：将精炼片段组装为结构化上下文 $C_A$，送入 Prompt 模板。
\end{enumerate}

\subsection{语义抽取与规范化验证流程}

LLM 的输出被强制要求符合预设的 JSON 格式，包含：扰动行为自然语言描述、$T_{in}$ 可执行伪代码、$R_{out}^{\epsilon}$ 关系断言以及适用前提。为确保代码在投入真实执行前安全可用，必须经过三层规范化检查，详见算法 \ref{alg:app_mr_gen}。

\begin{algorithm}[htbp]
    \caption{应用层知识检索与语义关系抽取算法}
    \label{alg:app_mr_gen}
    \begin{algorithmic}[1]
        \Require 任务类型 $A$，知识库 $\mathcal{K}$，大语言模型 LLM
        \Ensure 应用层候选关系集合 $\mathcal{M}_{app}$
        \State $\mathcal{M}_{app} \leftarrow \emptyset$
        \State $C_A \leftarrow \text{RetrieveAndDenoise}(A, \mathcal{K})$
        \State $Prompt \leftarrow \text{BuildStructuredPrompt}(A, C_A)$
        \State $Candidates \leftarrow \text{LLM.GenerateJSON}(Prompt)$
        \For{each $c \in Candidates$}
        \If{\textbf{not} $\text{AST\_SyntaxCheck}(c.T_{in}, c.R_{out})$}
        \State \textbf{continue} \Comment{剔除语法非闭环或包含危险调用的代码}
        \EndIf
        \If{\textbf{not} $\text{TypeAndParamCheck}(c, A.\text{interface})$}
        \State \textbf{continue} \Comment{剔除维度或类型不匹配的接口调用}
        \EndIf
        \State $\mathcal{M}_{app} \leftarrow \mathcal{M}_{app} \cup \{(c.T_{in}, c.R_{out}^{\epsilon})\}$
        \EndFor
        \Return $\mathcal{M}_{app}$
    \end{algorithmic}
\end{algorithm}

需要强调的是，AST 校验仅负责“语法与调用的合法性拦截”，并不证明其语义在数学上必然成立。其语义合理性将交由后续的统计大样本校验完成。

\section{多阶段质量保障与统一入库机制}

三个层次生成的 MR 集合 $\mathcal{M}_{op}, \mathcal{M}_{model}, \mathcal{M}_{app}$ 被统一汇入全局调度总线，执行统一的多阶段质量保障（QA）流水线，剔除劣质与错误规则。

\subsection{沙箱预筛选（运行时基础过滤）}

第一阶段通过动态试运行剔除明显的逻辑崩塌。
\begin{itemize}
    \item \textbf{采样策略}：基于被测对象的输入空间约束，使用边界生成器随机采样 $K=5$ 组极小规模合法张量。
    \item \textbf{剔除条件}：在隔离沙箱中执行 $MR(x)$。若触发编译崩溃（如算子后端未实现）、API 调用越界（如张量维度超出限制）或输出判定在 5 次运行中全盘失败，则直接将该 MR 剔除。
\end{itemize}

\subsection{分流校验：形式化证明与统计置信度验证}

通过预筛选的 MR 将根据其特征被分流至强约束校验或统计校验路径：
\begin{itemize}
    \item \textbf{符号化形式校验（SymPy Proof）}：专门针对算子层中不包含复杂控制流的纯数学关系（如线性映射、加法交换）。将 Python 逻辑转换为代数表达式 $E(x)$，利用引擎求解 $simplify(LHS - RHS) \equiv 0$。通过该验证的 MR 具备在给定形式系统内可验证的正确性。
    \item \textbf{经验置信度验证}：针对模型层的图结构约束关系、应用层的语义抽取关系以及部分无法形式化的复杂算子（如 \texttt{ReLU} 分段函数）。在参考基准环境上抽取规模 $N \ge 100$ 的样本集，统计满足预设验证关系的通过率 $\hat{p}_{MR}$。仅当 $\hat{p}_{MR} \ge \theta$（经验阈值设为 $0.95$）时，方认定该规则具备合理性。
\end{itemize}

\subsection{统一关系库入库}

通过流水线任意一侧校验的 MR 将被统一写入 SQLite 关系数据库。入库时系统严格记录其元数据，包括：来源层次（\texttt{Op/Model/App}）、验证标签（\texttt{SymPy\_Verified / Stat\_Verified}）、依赖的特定约束前提以及版本生成信息。这种规范化的元数据结构，使得后续测试引擎在提取调度时，可以实现精确的靶向匹配，最终完成了“覆盖生成、校验、执行与入库的完整测试流程”。

\section{本章小结}

本章详细阐述了面向深度学习框架的分层蜕变测试方法，证明了算子层、模型层与应用层并列构成了一个完整的方法体系。本章首先统一了分层测试的形式化基础与验证范式；随后，通过定义详尽的伪代码算法链路，依次实现了算子层的模板精准推导、模型层的受控约束模式匹配以及应用层的闭环语义抽取。最后，设计了覆盖代码预筛、符号证明与置信验证的统一质量保障与入库机制。本章所建立的一切对象抽象、算法流程与约束规则，将在第4章转化为 DeepMT 系统中真实可执行的工程模块代码。

第3章建立了面向深度学习框架的分层蜕变测试方法体系。为验证上述理论与方法的工程有效性，本章将其具体实现为一个自动化的测试框架原型系统——DeepMT（Deep Learning Metamorphic Tester）。本章的重点在于阐述系统核心模块的具体实现方式、数据流转逻辑以及各模块的工程边界。

\section{系统需求分析与总体架构}

\subsection{实现边界与总体说明}

在工程落地中，DeepMT 系统在算子层、模型层和应用层均实现了对应的生成与执行机制。三层机制在微内核架构下共享统一的中间表示（IR）、入库调度与验证逻辑，但各层的支持对象、规则覆盖范围和验证场景依据其所在层次的特征存在显著差异。

本系统当前已完成的核心实现包括：覆盖三层的自动化蜕变关系（MR）生成底座、基于混合策略的数据采样模块、动态误差评估（Oracle）模块，以及针对 PyTorch、TensorFlow 和 PaddlePaddle 实验规定版本的执行适配器。本文后续的实验评估均基于上述已完成的代码模块展开。

\subsection{功能需求与非功能需求}

面向深度学习框架的分层测试需求，DeepMT 原型系统需满足以下条件：

\textbf{功能需求：}
\begin{itemize}
    \item \textbf{多层次 MR 生成与管理：} 系统需实现面向算子、模型、应用三层的生成引擎，支持形式化推导、计算图结构匹配及大语言模型语义抽取，并利用 SQLite 提供 MR 的持久化管理。
    \item \textbf{动态测试调度与执行：} 系统需根据 IR 动态生成边界值与多维随机组合的测试张量，驱动底层目标框架执行源输入与变换输入。
    \item \textbf{跨框架适配支持：} 系统需提供统一的执行层抽象，在本文实验范围内支持对不同异构深度学习后端（含 PyTorch, TensorFlow, PaddlePaddle）的统一调用。
    \item \textbf{闭环违例检测与报告生成：} 系统需基于自适应容差机制完成输出一致性评估，捕获违例后自动执行缺陷初步分类并导出结构化报告。
\end{itemize}

\textbf{非功能需求：}
\begin{itemize}
    \item \textbf{高可扩展性：} 遵循微内核架构设计，新增框架适配或 MR 规则应通过注册独立插件或配置文件实现，不修改核心控制流。
    \item \textbf{执行可复现性：} 系统的调度执行必须实现全局随机种子（Seed）控制与沙箱环境快照，以支持缺陷执行轨迹的可重复验证。
\end{itemize}

\subsection{总体架构设计与微内核机制}

DeepMT 采用了“微内核 + 插件化（Microkernel \& Plugin）”架构模式，将测试框架的调度控制流与易变的基础设施依赖进行了工程解耦。

系统的代码组织被划分为三个主切面：
\begin{enumerate}
    \item \textbf{微内核（Core Engine）：} 包含任务调度器 \texttt{TaskScheduler} 与数据抽象中心 \texttt{IRManager}。微内核掌握系统事件流转，不包含具体深度学习框架的显式依赖。
    \item \textbf{多源生成引擎（Generation Engines）：} 包含平行的三层生成模块，负责输出通过质量校验的 \texttt{MRRecord}。
    \item \textbf{框架适配插件层（Plugin Layer）：} 异构计算适配器（Adapters）作为动态挂载的插件存在。通过实现 \texttt{FrameworkAdapter} 协议，系统在运行时按需加载目标框架。
\end{enumerate}

\subsection{算子层生成引擎实现}
\texttt{OperatorMRGenerator} 负责处理算子级 IR。
\begin{itemize}
    \item \textbf{输入与输出：} 接收 \texttt{OperatorIR}，输出符号化证明通过的 \texttt{MRRecord} 列表。
    \item \textbf{实现逻辑：} 引擎加载 JSON 格式的数学性质模板库。识别算子后，通过 Jinja2 模板将抽象规则（如乘法分配律）渲染为 Python 匿名函数。随后调用 \texttt{SymPyProver} 内部的 AST 解析器将其映射为 \texttt{sympy.MatrixSymbol} 并执行 \texttt{simplify()}。仅当表达式结果为 $0$ 时，该记录被放行入库。
\end{itemize}

\subsection{模型层生成引擎实现}
\texttt{ModelMRGenerator} 聚焦于计算图的拓扑结构分析。
\begin{itemize}
    \item \textbf{输入与输出：} 接收表示模型结构的 \texttt{ModelIR}，输出图变换级别的 \texttt{MRRecord}。
    \item \textbf{实现逻辑：} 依赖 \texttt{GraphAnalyzer} 组件将模型导出为 ONNX 格式的有向无环图（DAG）。\texttt{PatternMatcher} 遍历 DAG，若匹配到预设模式（例如连续卷积层后接全局平均池化），则实例化平移不变性 MR。图同构检测算法将在此阶段验证变换前后的图结构合法性。
\end{itemize}

\subsection{应用层生成引擎实现}
\texttt{ApplicationMRGenerator} 集成大语言模型（LLM）API 进行语义特征抽取。为缓解不可控的风险，系统设计了严格的校验回退机制。
\begin{itemize}
    \item \textbf{输入与输出：} 接收 \texttt{ApplicationIR}（含官方接口文档摘要），输出语义扰动策略。
    \item \textbf{实现逻辑：} \texttt{LLMPromptBuilder} 将任务元数据与固定 Few-Shot 模板组装，调用大语言模型生成 JSON 格式建议。
    \item \textbf{校验与回退：} 引擎内置 \texttt{SchemaValidator}。LLM 输出必须严格符合预定义的 JSON Schema。若 Schema 解析失败或包含未注册的扰动函数，系统将启用重试机制；若仍失败，则回退触发默认的基线语义 MR。
\end{itemize}

\subsection{关系知识库存储结构}
生成的 MR 统一经由 \texttt{MRKnowledgeBase} 组件持久化至 SQLite 数据库。核心数据表支持后续测试调度的快速检索，包括哈希主键、所属层次、变换代码与验证元数据等字段。

\subsection{实验环境与测试对象设定}

\begin{itemize}

\section{研究工作总结}

深度学习框架作为支撑人工智能生态的基础设施，其底层代码的质量与数值稳定性直接决定了上层应用模型的可靠性。然而，由于框架底层计算的非确定性、浮点误差的累积特征以及跨层语义的复杂性，传统依赖精确预期输出的测试方法面临严重的“测试预言机”失效困境。针对这一突出的软件工程痛点，本文以蜕变测试理论为核心，开展了面向深度学习框架的分层蜕变测试方法研究与自动化原型系统实现工作。

在整个研究与工程实践周期内，本文紧密围绕预言机问题，完成了一个完整的“方法设计—算法实现—系统原型—实验验证”闭环，主要取得了以下实质性成果：

第一，构建了面向深度学习框架的分层蜕变测试总体方案。针对现有测试工作碎片化、缺乏系统性视角的不足，本文将复杂的框架软硬件栈在逻辑上解耦为算子层、模型层和应用层三个同级测试维度。通过明确各层的测试边界与失效特征，确立了“底层关注数值一致性、中层关注拓扑结构不变性、高层关注语义稳定性”的统一分层测试思路。同时，设计了统一中间表示（IR）与跨框架适配机制，为异构环境下的通用测试调度奠定了数据结构基础。

第二，提出并实现了面向分层对象的多源融合蜕变关系自动生成与质量保障方法。针对传统蜕变测试中人工构造 MR 效率低下的瓶颈，本文在三个抽象层次上设计了平行的生成策略：在算子层，通过形式化规约与数学性质模板匹配实现了高精度的 MR 提取；在模型层，基于计算图静态结构分析实现了拓扑模式到 MR 的结构化实例化；在应用层，引入大语言模型（LLM）与检索增强技术实现了领域语义 MR 的知识抽取。在此基础上，构建了由沙箱预筛选、符号化形式证明与统计置信度评估组成的三阶段流水线，保障了生成结果的有效性。

第三，设计并落地了自动化测试框架原型系统 DeepMT。作为承载上述理论的工程实体，DeepMT 系统采用了微内核与插件化的架构设计，实现了核心调度与底层框架 API 的解耦。系统真实打通了端到端闭环，并在本文实验范围内完整接入了 PyTorch、TensorFlow 与 PaddlePaddle 三大主流框架。

第四，依托原型系统开展了系统化的实证评估与缺陷挖掘。实验引入了 Manual-MR、LLM-Only 与 Template-Only 作为科学对比基线。数据表明，相较于纯人工构造数小时级的单条产出耗时，本系统实现了秒级的高效生成；相较于单源方法，多源融合策略结合多阶段校验，将各层蜕变关系的有效留存率大幅提升（最高达 73.3\%，最终有效留存总计 363 条高质量资产）。在受控变异实验中，相较于纯模板基线，DeepMT 将各层的变异检出率平均提升了 20\% 至 40\%，在算子层取得了 81.25\% 的高检出率。此外，系统在原生框架中成功复现了 11 处真实的边缘计算缺陷，并跨框架量化了底层实现分歧。上述实证结果以定量数据的形式，充分证实了本文方法的可行性与卓越的工程价值。
